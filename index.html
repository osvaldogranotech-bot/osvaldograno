Route Target mismatch — VPNv4 BGP / L3VPN
Financial services — Bank of America

A large financial institution reported complete loss of routing between two branch locations. Given the customer's regulatory environment, restoration had to be achieved without dropping any active BGP sessions. Starting at the PE, the remote site's prefix was absent from the VRF routing table. The VPNv4 BGP table showed the route was being received from the far-end PE but failing to install — traced to a Route Target import mismatch introduced during a recent change window. Corrected the RT import policy, issued a soft BGP reset to avoid session disruption, and confirmed the prefix populated in both the VPNv4 table and VRF routing table without any additional session impact.

BGP
Site-to-site reachability failure across agency WAN
CPE BGP session idle — keepalive timer mismatch post-upgrade
Federal government — SSA / NOAA

A federal agency reported intermittent loss of connectivity from field offices to their primary data center — a high-visibility issue affecting mission-critical operations. The PE VRF routing table was intact, but a branch prefix had gone missing. Accessing the customer CPE at the affected site revealed the eBGP session to the PE had gone idle following a CPE software upgrade that reset keepalive and hold-timer values to non-matching defaults. Coordinated the timer correction with the agency's team, re-established the session, and confirmed the prefix repopulated in both the PE's VRF and VPNv4 tables. Full reachability validated with VRF-scoped pings before ticket closure.

LDP
MPLS label path black-holing store traffic
LDP re-sync failure after link flap — P core router
Retail — Discount Tire (multi-site)

A national retail chain with hundreds of store locations reported one-way traffic loss affecting point-of-sale systems across multiple sites. BGP and VRF tables on both PEs appeared healthy, making this harder to isolate. Using MPLS ping and traceroute toward the remote PE loopback identified a black hole at a P core router where an LDP session had partially recovered after a link flap but had not fully re-synced its label bindings — the LFIB was advertising stale label entries. Confirmed the mismatch, coordinated with the backbone team to clear the LDP session, monitored label re-convergence, and restored bidirectional traffic to all affected store POS systems.

QoS
Voice call quality degradation across clinic WAN
DSCP mis-marking after UC platform migration
Healthcare — Walgreens health clinics

A national pharmacy and health clinic operator reported severe voice quality issues across their MPLS WAN following a unified communications platform migration. Bandwidth was within normal limits, pointing to a QoS classification problem. Checking the service policy on the PE's CE-facing interface revealed voice RTP traffic was being placed into the default best-effort class — the new UC platform was marking voice at DSCP 46 but using a non-standard mark for signaling that the existing class-map did not match. Updated the class-map on the PE, verified traffic was queuing correctly in the EF class via interface policy counters, and confirmed with the customer that call quality returned to expected levels across all clinic locations.

Lab topology

A full MPLS L3VPN lab replicating the AT&T service model — CE routers peering eBGP to PE routers, PEs exchanging VPNv4 routes over iBGP via loopbacks, IS-IS + LDP running across the P core, and an ASBR for peering simulation.

  • 01P router — IS-IS + LDP only, verify label exchange
  • 02PE routers — add VRF, configure VPNv4 iBGP via loopbacks
  • 03CE routers — eBGP to PE, advertise customer LAN prefixes
  • 04Verify E2E — VRF ping, MPLS traceroute, VPNv4 table
  • 05Inject faults — simulate RT mismatch, LDP flap, BGP timer issue
  • 06ASBR — add eBGP peering scenario for ISP simulation
